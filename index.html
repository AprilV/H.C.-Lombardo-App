What went well?

Sprint 12 was completed on time — slightly ahead of schedule. 8 of 9 tasks delivered. The one incomplete task (TA-008) was blocked by an external dependency outside the team's control.

  • Dashboard built, deployed, and live — professional quality suitable for employer and professor review
  • Product backlog, sprint backlog, and agile structure fully established with correct terminology (task/subtask — no "issue")
  • All 22 API endpoints tested — EC2, PostgreSQL, Amplify, and CORS all verified operational
  • Real problems discovered during audit and logged to backlog (TA-052 through TA-056) — not ignored, not fixed out of scope
  • Sprint execution process documented — section-by-section workflow with deploy after every section
  • Task resolution modal built — click any task to see what was done, when, and what was logged as a result
  • Project published to Handshake AI Showcase — professional visibility achieved
  • Production meta description corrected — accurate representation of the project
  • Communication style refined — screenshot-by-screenshot review, one task per session, quality over quantity established as the standard
  • Persistent memory improved — operating procedures, terminology rules, and sprint process all documented and saved
What didn't go well?

Several recurring friction points slowed progress and required correction. Most stemmed from the AI moving too fast and not consistently following documented process.

  • AI made unauthorized changes (Open Graph tags added without being asked — required a git revert)
  • Wrong terminology used repeatedly — "issue" instead of "task" — had to be corrected multiple times across the same session
  • Chart updates were batched or skipped — burndown, velocity, burnup, and severity donut had to be caught and corrected by PM
  • AI read process documents at session start but drifted mid-session — same rules had to be re-explained multiple times
  • Changes pushed directly to GitHub (production) without local testing or PM approval — no verification step in place
  • AI asked questions that could have been answered by reading the code rather than asking the PM
  • Some sessions lost alignment — AI got confused or pushed forward when PM needed to pause and verify
  • Future sprint slots were pre-assigned items that had to be removed — violates agile process
  • Risk register and some dashboard sections not kept current — PM had to catch omissions
What will we do differently next sprint?

Improvements are centered on discipline, process adherence, and treating GitHub as production — because it is.

  • Quality over quantity — slow down, verify each step before moving forward, never rush to the next section
  • PM approval before any push — no change goes to GitHub without April's sign-off; master and gh-pages are production
  • Local verification before every push — test the change locally, confirm it works, then ask for push approval
  • AI reviews process docs before each section — not just at session start; refer back to SPRINT_EXECUTION_PROCESS.md and memory throughout
  • Ask permission, not forgiveness — if something wasn't explicitly requested, ask before doing it
  • One task per session — maintain the discipline established in S12; no scope creep within a session
  • Continue testing AI persistent memory — April will verify that process rules are being followed without re-explanation
  • Identify additional risks — risk register review at sprint start to capture anything discovered in S12 that wasn't logged
  • Terminology holds — task, subtask, blocker; no deviation; if AI uses "issue" incorrectly, correct immediately
  • Sprint process is documented — AI has no excuse for not following it; the documentation exists and is saved to memory
What went well?

Sprint 13 execution stabilized and closeout quality improved. Core data/ML cleanup goals were completed, governance alignment was tightened, and dashboard metrics were normalized to reflect real actionable scope.

  • Completed key Sprint 13 tickets including TA-057, TA-058, TA-059, TA-067, TA-068, TA-069, TA-010, TA-012, and TA-013
  • PM dashboard source-of-truth transition and documentation alignment were finalized (DSH-011 moved to DONE)
  • Sprint 13 progress math was corrected to exclude deferred backlog carryover, resulting in accurate effective completion metrics
  • Governance logs, sprint status text, and chart narratives were synchronized to the same validated Sprint 13 state
  • Hours log and dashboard evidence were updated to support advisor-facing closeout review
What didn't go well?

A few recurring friction points created status confusion and delayed clean sprint-close confidence, mostly around external dependencies and dashboard denominator drift.

  • TA-011 subtask s11_2 remained blocked due EC2/SSH accessibility constraints outside normal coding flow
  • Deferred TA-063 subtasks initially stayed in Sprint 13 denominator, causing misleading completion percentages until corrected
  • Large single-file dashboard maintenance increased noise risk when validating exact sprint counts and narratives
  • Closeout readiness messaging lagged behind actual ticket state and required governance text cleanup
What will we do differently next sprint?

Sprint 14 will use tighter close-control rules so progress, blocker carryover, and governance messaging stay aligned from day one.

  • Apply deferred-task denominator policy at the time of deferment, not during end-of-sprint cleanup
  • Keep one blocker carryover list that is explicitly separated from completed and deferred scope
  • Run sprint-count verification script before each governance/status update commit
  • Update retrospective, DoD, and release-note evidence in the same closeout pass before sprint flip
  • Maintain one-task/one-subtask execution discipline with explicit checkpoint confirmation before moving on

Application Features
📊
Historical Analytics
7,263 games from 1999–2025. EPA metrics, weather impact, rest advantage, referee tendencies, and ATS betting performance.
🤖
ML Predictions
XGBoost winner classifier + spread regressor using 46 rolling features. Elo ratings system. Ensemble predictions vs. Vegas lines.
📡
Live Scores
Real-time game data via ESPN API, overlaid with AI predictions, Elo ratings, and Vegas spread comparisons.
🏈
Team Tools
Head-to-head team comparison, individual team detail pages, matchup analyzer, and custom stat builder.
📈
Model Performance
Season-long tracking of AI predictions vs. actual results vs. Vegas. Accuracy metrics, MAE, and ATS record.
⚙️
Admin Panel
Data refresh controls, background updater management, and database health monitoring.
